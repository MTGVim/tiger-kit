# gaplan

## 목적

`.gap/{branch_name}/analysis/gap-report.md`를 검토 가능한 구현 계획 초안으로 바꿉니다.

사용자가 계획 수립, 작업 분해, 순서 정하기, 검증 설계, 계획 모드 스타일의 구현 준비를 원할 때 이 스킬을 사용합니다.

## 동작 방식

사용자에게 보이는 응답과 TIGAP 산출물은 항상 한글로 작성합니다. 인용한 원문, 코드, 명령어, 파일 경로, 식별자는 원문 그대로 유지할 수 있습니다.

구현보다 계획을 우선합니다.

사용자가 같은 단계에서 구현까지 명시적으로 요청하지 않는 한 이 스킬에서는 프로덕션 코드를 수정하지 않습니다.

사용자가 구현 계획을 검토하고 승인하기 전에는 실행 작업 목록을 확정하지 않습니다.

## 필수 입력

다음 파일을 읽습니다.

```text
.gap/{branch_name}/normalized/source-packet.md
.gap/{branch_name}/analysis/gap-report.md
```

파일이 없다면 사용자에게 다음 명령을 실행하라고 요청합니다.

```text
/tigap:gap
```

또는 먼저 최소한의 자료 수집을 수행합니다.

기존 산출물의 브랜치/작업 ID를 유지합니다. 현재 브랜치가 `main`, `master`, `develop` 또는 저장소 기본 브랜치이고 새 계획 산출물을 쓰려는 경우, 원천 자료 전용 작업 브랜치나 명시적 작업 ID로 계속할지 권유합니다. 사용자 승인 없이 브랜치 생성이나 전환을 수행하지 않습니다.

## 필수 출력 파일

먼저 다음 파일을 생성하거나 갱신합니다.

```text
.gap/{branch_name}/plan/implementation-plan.md
```

사용자가 계획을 검토하고 승인한 뒤에만 다음 파일을 생성하거나 갱신합니다.

```text
.gap/{branch_name}/tasks.md
```

필요하면 다음 파일도 생성합니다.

```text
.gap/{branch_name}/review-checklist.md
```

## 계획 원칙

1. 작업을 작게 유지합니다.
2. 수정 전에 먼저 확인합니다.
3. 구현 전에 변경 가능성이 있는 파일을 식별합니다.
4. 작업마다 검증 방법을 정의합니다.
5. 가정과 확인된 요구사항을 분리합니다.
6. 추측하지 말고 차단 사유를 표시합니다.
7. 명확한 이유가 없다면 기존 프로젝트 패턴을 유지합니다.

## 진행 절차

### 1. 갭 산출물 읽기

source packet과 갭 분석 보고서를 읽습니다.

다음을 추출합니다.

- 목표
- 범위
- 논의 범위에서 제외할 것
- 가정
- 위험
- 확인이 필요한 질문
- 구현 갭
- 검증 필요 사항

### 2. 코드 구조 확인

저장소를 사용할 수 있다면 관련 디렉터리와 주변 패턴을 확인합니다.

다음을 살펴봅니다.

- 아키텍처 경계
- 컴포넌트 소유 범위
- 데이터 가져오기 흐름
- 상태 관리
- 검증 계층
- 테스트 관례
- 네이밍 관례
- feature flag 패턴

### 3. 구현 계획 작성

`templates/implementation-plan.md`를 사용합니다.

각 작업에는 다음을 포함합니다.

- 목적
- 확인할 파일
- 변경 가능성이 있는 파일
- 구현 메모
- 검증 방법
- 위험

### 4. 계획 검토 대기

구현 계획 초안을 작성한 뒤 사용자 검토를 기다립니다.

검토가 끝나기 전에는 실행 작업 목록을 확정하지 않습니다.

사용자가 계획을 승인하면 `templates/tasks.md` 형식에 맞춰 작업 목록을 작성합니다.

작업은 하나씩 실행할 수 있어야 합니다.

좋은 작업은 `/tigap:go`가 범위를 잃지 않고 완료할 수 있을 만큼 작습니다.

### 5. 준비 상태 검토

마무리하기 전에 다음을 확인합니다.

- 모든 요구사항이 작업 또는 명시적 non-goal에 매핑되었는가?
- 위험이 큰 영역마다 검증 방법이 있는가?
- 차단 사유가 명확히 표시되었는가?
- 첫 작업이 안전하고 작은가?

## 완료 기준

계획 초안 작성 단계는 다음 조건을 만족하면 완료됩니다.

- 구현 계획이 존재함
- 가정, 위험, 검증 방법이 정리됨
- 사용자가 검토할 결정 사항이 명확함
- 작업 목록은 아직 확정하지 않음

계획 검토 후 작업 목록 작성 단계는 다음 조건을 만족하면 완료됩니다.

- 작업 목록이 존재함
- 작업 순서가 정해짐
- 의미 있는 각 작업에 검증 방법이 붙어 있음
- 큰 재계획 없이 `/tigap:go`를 시작할 수 있음

## 인계

계획 초안이 준비되면 먼저 사용자 검토를 요청합니다.

계획이 승인되고 작업 목록이 준비된 뒤에만 다음 명령을 추천하며 마무리합니다.

```text
/tigap:go
```
