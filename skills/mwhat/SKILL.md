# mwhat

## 목적

긴 LLM 답변이나 애매한 설명을 다시 장황하게 요약하지 않고, 사용자가 바로 판단하거나 답할 수 있는 짧은 한국어 해설로 바꿉니다.

핵심 질문은 이것입니다.

```text
뭣? 그래서 이 말이 무슨 뜻이고, 나는 지금 어떻게 답하면 되지?
```

## 자연어 트리거

사용자가 명령어를 정확히 쓰지 않아도 아래처럼 “못 알아듣겠다”는 반응이면 `mwhat`으로 처리합니다.

- `뭐라고?`
- `뭐라는거야?`, `뭐라는 거야?`
- `무슨말이죠?`, `무슨 말이죠?`, `무슨 말임?`
- `뭣`, `뭣?`
- `그래서 뭐 하라는 거야?`
- `이 답변 무슨 뜻이야?`
- `아래 답변 좀 사람이 알아듣게 풀어줘`

이 트리거들은 새 요구사항을 만들거나 구현하라는 뜻이 아니라, 먼저 원문의 의도와 막힌 지점을 해독하라는 신호입니다.

## 동작 방식

사용자에게 보이는 응답은 기본적으로 한글로 작성합니다. 인용한 원문, 코드, 명령어, 파일 경로, 식별자는 원문 그대로 유지할 수 있습니다.

이 스킬은 구현, 파일 수정, 계획 수립을 하지 않습니다. 사용자가 제공한 답변이나 설명을 해석해서 실행 가능한 다음 말로 줄이는 데 집중합니다.

## 기본 출력 형식

기본 출력은 두 블록만 사용합니다.

```md
🤔 뭣? 쉽게 말하면
이 말은 A를 하자는 뜻이고, 지금은 B 기준이 없어서 바로 시작하기 어렵다는 말이에요.

💡 추천
이렇게 말하세요: 진행하려면 B 기준을 먼저 정해야 할 것 같습니다.
```

### 블록별 역할

- `🤔 뭣? 쉽게 말하면`: 원문이 실제로 말하는 뜻, 하던 것, 문제 상황을 최대 3줄로 해석합니다.
- `💡 추천`: `이렇게 말하세요: ...` 또는 `이렇게 하세요: ...` 형식으로, 사용자가 그대로 말하거나 바로 할 다음 행동을 최대 2줄로 제안합니다.

## 말투

친절하고 부드럽지만 단정적으로 말합니다. 딱딱한 보고서체보다 편한 해설자 톤을 사용하되, 결과는 업무 대화에 붙여도 어색하지 않아야 합니다.

선호 표현:

- `이 말은 ...라는 뜻이에요.`
- `핵심은 ...예요.`
- `지금 바로 할 수 없는 이유는 ...예요.`
- `추천은 ...입니다.`

피할 표현:

- `~에 가까워요`
- `아마도`를 남발하는 문장
- `일 수도 있어요`를 남발하는 문장
- 과한 역할극, 플러팅, 밈만 있는 문장
- 원문보다 길어지는 재요약

## 판단 규칙

1. 원문보다 훨씬 짧게 씁니다.
2. 원문에 없는 일을 만들어내지 않습니다.
3. 확실한 내용과 추정한 내용을 섞지 않습니다.
4. `🤔 뭣? 쉽게 말하면`은 뜻과 문제 상황을 합쳐 최대 3줄로 씁니다.
5. `💡 추천`은 따옴표로 감싼 대사체보다 `이렇게 말하세요: ...` 또는 `이렇게 하세요: ...` 형식으로 쓰고 최대 2줄로 제한합니다.
6. 원문이 이미 실행 가능하면 `🤔 뭣? 쉽게 말하면`에서 “바로 실행해도 되는 말”이라고 짧게 밝힙니다.
7. 원문에 실질 액션이 없으면 `💡 추천`에서 추가 질문이나 확인 문장을 제안합니다.
8. 기본 두 블록 외의 선택 블록은 사용하지 않습니다.

## 예시

입력:

```text
이 기능은 확장성을 고려해서 모듈화하고, 상태 관리는 나중에 요구사항이 명확해지면 정하는 게 좋겠습니다. 우선 데이터 흐름을 정리하고 컴포넌트 경계를 잡는 방향으로 접근하면 될 것 같습니다.
```

출력:

```md
🤔 뭣? 쉽게 말하면
이 말은 “지금 바로 구현하지 말고 구조부터 정하자”는 뜻이에요. 상태 관리는 아직 정하지 말고, 데이터 흐름과 컴포넌트 경계부터 잡자는 말입니다. 다만 컴포넌트 분리 기준과 상태 관리 결정 조건이 아직 없어서 바로 구현 범위를 잡긴 어려워요.

💡 추천
이렇게 말하세요: 좋습니다. 이번 단계에서는 데이터 흐름과 컴포넌트 경계를 먼저 정리하고, 상태 관리는 어떤 조건에서 결정할지도 같이 적어두겠습니다.
```

## 중단 조건

사용자가 실제 구현, 파일 수정, 테스트 실행을 요청하면 이 스킬만으로 진행하지 말고, 먼저 이 해석이 맞는지 짧게 확인합니다. `mwhat`은 실행 전에 말을 이해시키는 스킬입니다.
