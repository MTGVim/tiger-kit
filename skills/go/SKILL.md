# go

## 목적

`.gap/{branch_name}/tasks.md`의 작업을 한 번에 하나씩 실행합니다.

갭 분석과 계획 수립이 끝난 뒤 사용자가 구현을 시작하고 싶을 때 이 스킬을 사용합니다.

## 동작 방식

사용자에게 보이는 응답과 TIGAP 산출물은 항상 한글로 작성합니다. 인용한 원문, 코드, 명령어, 파일 경로, 식별자는 원문 그대로 유지할 수 있습니다.

작은 단계로만 실행합니다.

앞질러 진행하지 않습니다. 차단 사유나 모순을 발견하지 않는 한 계획을 다시 쓰지 않습니다.

## 필수 입력

다음 파일을 읽습니다.

```text
.gap/{branch_name}/plan/implementation-plan.md
.gap/{branch_name}/tasks.md
```

파일이 없다면 사용자에게 다음 명령을 실행하라고 요청합니다.

```text
/tigap:gaplan
```

또는 먼저 최소한의 계획을 작성합니다.

현재 브랜치가 `main`, `master`, `develop` 또는 저장소 기본 브랜치라면 코드 변경이나 산출물 갱신 전에 원천 자료 전용 작업 브랜치나 명시적 작업 ID로 계속할지 권유합니다. 사용자 승인 없이 브랜치 생성, 전환, push를 수행하지 않습니다.

## 필수 출력 파일

다음 파일을 갱신합니다.

```text
.gap/{branch_name}/tasks.md
.gap/{branch_name}/execution-log.md
```

필요하면 다음 파일도 갱신합니다.

```text
.gap/{branch_name}/review-checklist.md
.gap/{branch_name}/archive/manifest.md
```

Archive 파일 이동이나 정리가 필요하면 대상과 목적을 먼저 보고하고 사용자 승인을 받습니다.

## 진행 절차

### 1. 다음 작업 선택

사용자가 다른 작업을 지정하지 않았다면 `Ready`에 있는 첫 번째 미완료 작업 하나만 선택합니다.

여러 Ready 작업이 있어도 이번 실행에서는 선택한 작업 하나만 처리합니다.

일반 작업이 모두 끝났고 cleanup 작업가 남아 있다면 cleanup 작업를 마지막 작업으로 선택합니다.

해당 작업을 `In Progress`로 옮기거나 현재 진행 중임을 명확히 표시합니다.

### 2. 수정 전 확인

코드를 바꾸기 전에 다음을 확인합니다.

- 계획에 적힌 파일
- 인접한 구현
- 테스트
- 기존 패턴
- 관련 import와 export
- 현재 브랜치/작업 ID가 기반 브랜치 직접 작업을 유도하지 않는지

### 3. 좁게 구현

현재 작업에 필요한 변경만 수행합니다.

정확성을 위해 반드시 필요한 경우가 아니라면 기회주의적 리팩터링을 피합니다.

### 4. 검증

가능하면 작업별 검증을 실행합니다.

기존 프로젝트 명령을 우선합니다.

- typecheck
- lint
- unit tests
- integration tests
- build

명령 이름은 프로젝트에서 쓰는 원문을 유지합니다.

검증을 실행할 수 없다면 이유를 기록합니다.

### 5. cleanup 작업 처리

cleanup 작업를 실행할 때는 다음을 확인합니다.

- 남은 `Ready` 또는 `In Progress` 작업이 없는지
- 의도된 `Blocked` 작업만 남았는지
- archive 대상 원천 자료, 계획, 작업 목록, 실행 로그가 무엇인지
- archive 대상과 목적을 사용자에게 보고했는지
- 필요한 검증이 끝났는지

실제 파일 이동이나 삭제가 필요하면 사용자 승인 없이 수행하지 않습니다.

### 6. Git handoff 준비

작업 완료 후 변경 파일이 있으면 commit 제안을 준비합니다.

cleanup까지 끝났고 작업 브랜치가 준비되어 있으면 push와 PR 생성을 제안할 수 있습니다.

제안에는 다음을 포함합니다.

- commit 대상 요약
- 권장 commit 메시지 초안
- push 대상 브랜치
- PR 제목 초안
- PR 본문 초안: 요약, 검증 결과, 관련 이슈

commit, push, PR 생성은 사용자 승인 없이 실행하지 않습니다.

현재 브랜치가 `main`, `master`, `develop` 또는 저장소 기본 브랜치라면 push를 제안하기 전에 작업 브랜치나 작업 ID로 옮길 것을 먼저 권장합니다.

### 7. 산출물 갱신

`tasks.md`를 갱신합니다.

- 완료된 작업은 `Done`으로 이동
- 끝나지 않은 작업은 `In Progress` 또는 `Blocked`에 유지
- 새로 발견한 작업은 `Ready` 또는 `Blocked`에 추가

`execution-log.md`에 다음을 추가합니다.

- 작업 이름
- 변경한 파일
- 실행한 검증
- 결과
- 진행률: `Done`, `In Progress`, `Ready`, `Blocked`의 체크박스 작업 개수
- 다음 작업: 첫 `Ready` 또는 `In Progress` 작업
- commit, push, PR 제안 여부
- 위험 또는 후속 작업

## 중단 조건

다음 상황에서는 멈추고 보고합니다.

- 요구사항이 실제 구현과 충돌함
- 필요한 원천 자료가 없음
- 검증이 실패했고 수정 방법이 명확하지 않음
- 작업 범위를 늘려야 함
- 선택한 Ready 작업 하나를 넘는 리팩터링이나 추가 구현이 필요함
- 구현이 관련 없는 영역에 영향을 줌

## 진행률 보고

작업을 마칠 때마다 사용자에게 전체 진행률을 보고합니다.

진행률은 `tasks.md`의 체크박스 작업만 계산합니다.

- 완료: `Done` 섹션의 체크된 작업 수
- 진행 중: `In Progress` 섹션의 미완료 작업 수
- 남음: `Ready` 섹션의 미완료 작업 수
- 차단: `Blocked` 섹션의 미완료 작업 수

첫 `Ready` 작업이 있으면 다음 작업으로 보고합니다. `In Progress` 작업이 있으면 먼저 이어서 끝낼 작업으로 보고합니다.

## 완료 기준

이 스킬은 작업 하나가 구현되고 검증되었거나 차단 사유가 명확히 문서화되고 진행률이 보고되면 완료됩니다.
