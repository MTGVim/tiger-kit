`.gap/feature-login/analysis/gap-report.md`가 이미 존재한다는 전제로, 승인 전 검토용 구현 계획 초안을 준비했습니다. 아직 승인 전이므로 `.gap/feature-login/tasks.md` 작업 목록은 확정하지 않습니다.

# 구현 계획 초안: feature-login

## 목표

- `feature-login` 갭 분석 보고서에서 식별된 로그인 기능 누락 사항을 구현 가능한 계획으로 정리합니다.
- 인증 흐름, 입력 검증, 오류 처리, 세션/토큰 처리, UI 상태 전환, 검증 전략을 구현 전 단계에서 명확히 합니다.
- 계획 승인 전에는 실행 작업 목록을 확정하지 않고, 검토가 필요한 결정 사항을 분리합니다.

## 범위

- 로그인 화면 또는 로그인 진입점 확인
- 사용자 자격 증명 입력 처리
- 클라이언트 측 입력 검증
- 로그인 API 호출 또는 기존 인증 서비스 연동
- 성공 시 인증 상태 저장 및 후속 화면 이동
- 실패 시 사용자에게 적절한 오류 표시
- 로딩, 중복 제출 방지, 네트워크 오류 처리
- 관련 단위/통합/수동 검증 계획 수립

## 논의 범위에서 제외할 것

- 신규 회원가입 기능
- 비밀번호 재설정 기능
- 소셜 로그인 추가
- MFA/2FA 신규 도입
- 권한/역할 기반 접근 제어의 전면 재설계
- 백엔드 인증 시스템의 대규모 변경
- 디자인 시스템 전면 개편

단, 갭 보고서에서 위 항목 중 일부가 로그인 완성에 필수로 명시되어 있다면 별도 결정 사항으로 재검토해야 합니다.

## 가정

- 기존 프로젝트에 인증 관련 API, 서비스, 라우팅, 상태 관리 패턴이 이미 일부 존재합니다.
- `feature-login`의 주요 갭은 로그인 흐름 연결, 검증, 오류 처리, 상태 반영, 테스트 보강에 있습니다.
- 현재 단계는 계획 초안 작성이며, 구현 코드는 수정하지 않습니다.
- 실제 변경 파일은 코드 구조 확인 후 확정해야 합니다.

## 의존성

- 기존 인증 API 계약
- 사용자 모델 또는 세션/토큰 저장 방식
- 라우팅/페이지 전환 방식
- 전역 상태 관리 또는 auth context/store 패턴
- 폼 검증 유틸리티 또는 스키마 검증 라이브러리
- 테스트 프레임워크와 기존 테스트 관례

## 현재 구현 요약

갭 보고서가 있다고 가정하면, 계획 수립 시 다음 항목을 기준으로 현재 구현을 확인해야 합니다.

- 로그인 UI가 존재하는지, 또는 새로 연결해야 하는지
- 로그인 폼이 API 호출까지 연결되어 있는지
- 인증 성공 결과가 앱의 인증 상태에 반영되는지
- 인증 실패, 검증 실패, 네트워크 실패가 구분되어 처리되는지
- 로그인 후 리다이렉트 규칙이 정의되어 있는지
- 이미 로그인한 사용자의 로그인 페이지 접근 동작이 정의되어 있는지
- 자동화 테스트가 현재 흐름을 보호하고 있는지

## 작업 분해 초안

아래는 승인 전 계획용 분해입니다. 아직 `/tigap:go`에서 실행할 확정 작업 목록은 아닙니다.

### 작업 1: 기존 인증 흐름과 경계 확인

- 목적: 구현 전에 로그인 관련 진입점, 상태 관리, API 경계, 라우팅 패턴을 확인합니다.
- 확인할 파일:
  - 로그인 페이지 또는 컴포넌트 파일
  - 인증 API 클라이언트 또는 service 파일
  - auth store/context/provider 파일
  - 라우터 또는 route guard 파일
  - 기존 인증/폼 테스트 파일
- 변경 가능성이 있는 파일:
  - 없음 또는 최소한의 계획 보완만 예상
- 구현 메모:
  - 기존 패턴을 우선 유지합니다.
  - 새 인증 구조를 만들기 전에 현재 프로젝트의 auth 책임 분리를 확인합니다.
- 검증 방법:
  - 로그인 요청부터 인증 상태 반영까지의 흐름을 문서상으로 추적합니다.
  - 갭 보고서의 각 요구사항이 기존 코드 위치에 매핑되는지 확인합니다.
- 위험:
  - 인증 상태 저장 위치를 잘못 판단하면 중복 상태 또는 라우팅 버그가 생길 수 있습니다.

### 작업 2: 로그인 폼 입력 검증 계획

- 목적: 이메일/아이디, 비밀번호 등 필수 입력의 검증 규칙과 사용자 피드백 방식을 정합니다.
- 확인할 파일:
  - 로그인 폼 컴포넌트
  - 공통 form/input 컴포넌트
  - 검증 유틸리티 또는 schema 파일
  - i18n/message 상수 파일이 있다면 관련 파일
- 변경 가능성이 있는 파일:
  - 로그인 폼 컴포넌트
  - 검증 schema 또는 form helper
  - 오류 메시지 상수
- 구현 메모:
  - 제출 전 필수값 검증을 수행합니다.
  - 기존 프로젝트의 validation 방식이 있으면 새 방식을 도입하지 않습니다.
  - 검증 메시지는 서버 오류 메시지와 구분합니다.
- 검증 방법:
  - 빈 값 제출 시 API 호출이 발생하지 않는지 확인합니다.
  - 잘못된 형식 입력 시 적절한 메시지가 표시되는지 확인합니다.
- 위험:
  - 서버 검증과 클라이언트 검증 메시지가 불일치할 수 있습니다.

### 작업 3: 로그인 API 연동 계획

- 목적: 로그인 제출 시 기존 API 계약에 맞춰 인증 요청을 보내고 결과를 처리합니다.
- 확인할 파일:
  - API client 설정 파일
  - auth service/login function
  - network error handling 유틸리티
  - 타입 정의 파일
- 변경 가능성이 있는 파일:
  - auth service 또는 API client wrapper
  - 로그인 폼 제출 핸들러
  - 인증 응답 타입 정의
- 구현 메모:
  - API 계약이 불명확하면 구현 전에 요청/응답 필드를 확인해야 합니다.
  - 성공, 인증 실패, 네트워크 실패를 분리해 처리합니다.
  - 중복 제출 방지를 위해 로딩 상태를 사용합니다.
- 검증 방법:
  - 정상 자격 증명으로 성공 응답 처리 확인
  - 잘못된 자격 증명으로 실패 메시지 확인
  - 네트워크 오류 또는 5xx 응답 처리 확인
- 위험:
  - 토큰 저장 방식이나 쿠키 기반 인증 여부가 불명확하면 구현이 막힐 수 있습니다.

### 작업 4: 인증 상태 저장 및 라우팅 계획

- 목적: 로그인 성공 후 앱의 인증 상태를 일관되게 갱신하고 올바른 화면으로 이동합니다.
- 확인할 파일:
  - auth context/store/provider
  - route guard 또는 middleware
  - 로그인 페이지 route
  - 로그인 후 기본 landing route
- 변경 가능성이 있는 파일:
  - auth state 관리 파일
  - 라우팅 설정 파일
  - 로그인 컴포넌트
- 구현 메모:
  - 기존 세션 복원 방식과 충돌하지 않게 처리합니다.
  - 로그인 전 접근하려던 경로가 있으면 redirect back 정책을 확인합니다.
  - 이미 로그인한 사용자가 로그인 화면에 접근했을 때의 동작을 정합니다.
- 검증 방법:
  - 로그인 성공 후 기대 경로로 이동하는지 확인합니다.
  - 새로고침 후 인증 상태가 유지 또는 복원되는지 확인합니다.
  - 보호된 페이지 접근 흐름이 정상인지 확인합니다.
- 위험:
  - 라우팅 변경은 다른 보호 페이지 접근 흐름에 영향을 줄 수 있습니다.

### 작업 5: 오류 표시와 접근성 계획

- 목적: 사용자가 로그인 실패 원인을 이해할 수 있도록 오류와 상태를 명확히 표시합니다.
- 확인할 파일:
  - 로그인 UI 컴포넌트
  - 공통 alert/error 컴포넌트
  - 접근성 관련 테스트 또는 lint 규칙
- 변경 가능성이 있는 파일:
  - 로그인 컴포넌트
  - 오류 메시지 컴포넌트
  - 테스트 파일
- 구현 메모:
  - 인증 실패와 시스템 오류를 구분합니다.
  - 로딩 중 버튼 비활성화 또는 진행 상태를 표시합니다.
  - 오류 메시지는 스크린 리더가 인지할 수 있는 방식으로 표시하는 것이 좋습니다.
- 검증 방법:
  - 실패 응답 시 오류 메시지가 표시되는지 확인합니다.
  - 재입력 또는 재시도 시 오류 상태가 적절히 초기화되는지 확인합니다.
  - 키보드만으로 로그인 흐름을 완료할 수 있는지 확인합니다.
- 위험:
  - 보안상 과도하게 구체적인 인증 실패 사유를 노출하면 안 됩니다.

### 작업 6: 테스트와 회귀 검증 계획

- 목적: 로그인 핵심 경로와 주요 실패 경로를 자동화 또는 수동 검증으로 보호합니다.
- 확인할 파일:
  - 기존 auth 관련 테스트
  - component 테스트 설정
  - API mocking 설정
  - e2e 테스트가 있다면 login spec
- 변경 가능성이 있는 파일:
  - 로그인 컴포넌트 테스트
  - auth service 테스트
  - route guard 테스트
  - e2e 로그인 테스트
- 구현 메모:
  - 프로젝트에 이미 있는 테스트 수준을 우선 따릅니다.
  - 모든 테스트를 한 번에 추가하기보다 핵심 경로부터 보호합니다.
- 검증 방법:
  - 성공 로그인 테스트
  - 필수값 누락 테스트
  - 인증 실패 테스트
  - 네트워크 오류 테스트
  - 로그인 후 라우팅 테스트
- 위험:
  - API mocking 방식이 기존 관례와 다르면 테스트가 취약해질 수 있습니다.

## 검증 계획

- 정적 검증:
  - 타입 체크가 있는 프로젝트라면 타입 검증 실행
  - lint 설정이 있다면 관련 lint 실행
- 자동화 테스트:
  - 로그인 컴포넌트 테스트
  - auth service/API wrapper 테스트
  - route guard 또는 인증 상태 테스트
  - 가능하면 로그인 성공/실패 e2e 테스트
- 수동 검증:
  - 빈 입력 제출
  - 잘못된 자격 증명 제출
  - 정상 로그인
  - 로그인 성공 후 이동 경로 확인
  - 새로고침 후 인증 상태 확인
  - 네트워크 오류 상황 확인

## 롤백 계획

- 로그인 UI 변경과 인증 상태 변경을 작게 나누어 적용합니다.
- API 연동 변경은 기존 auth service 경계를 유지해 되돌리기 쉽게 합니다.
- 라우팅 변경은 별도 커밋 또는 별도 작업으로 분리하는 것이 안전합니다.
- 문제가 발생하면 로그인 제출 핸들러와 라우팅 변경을 우선 되돌리고, 기존 인증 상태 관리 코드는 보존합니다.

## 사용자 검토가 필요한 결정 사항

1. 로그인 식별자는 이메일인가요, 사용자명인가요, 둘 다 허용하나요?
2. 인증 방식은 토큰 저장 방식인가요, httpOnly 쿠키 기반인가요?
3. 로그인 성공 후 이동할 기본 경로는 어디인가요?
4. 로그인 전 접근하려던 보호 페이지로 되돌아가야 하나요?
5. 인증 실패 메시지는 일반 메시지로 통일하나요, 실패 원인별로 구분하나요?
6. 이번 범위에 e2e 테스트 추가까지 포함하나요?
