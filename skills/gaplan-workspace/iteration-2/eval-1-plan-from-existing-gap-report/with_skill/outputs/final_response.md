## 구현 계획 초안

### 목표

`feature-login` 갭 분석 보고서에서 확인된 로그인 기능 요구사항과 현재 구현 간 차이를 안전하게 해소할 수 있도록 구현 범위, 변경 후보 파일, 검증 전략, 위험 요소를 정리합니다.

### 범위

- 로그인 입력/제출 흐름 보완
- 인증 API 연동 또는 기존 인증 호출부 정합성 확인
- 로그인 성공/실패 상태 처리
- 사용자 피드백 메시지 및 로딩 상태 처리
- 인증 결과 저장 또는 세션 갱신 방식 확인
- 기존 라우팅/리다이렉트 정책과의 정렬
- 관련 테스트 또는 수동 검증 절차 정의

### 제외 범위

- 회원가입, 비밀번호 재설정, 소셜 로그인 등 로그인 외 인증 기능 확장
- 인증 백엔드의 프로토콜 변경
- 전역 디자인 시스템 개편
- 권한 모델 또는 사용자 역할 체계 재설계
- 승인 전 `tasks.md` 확정 및 `/tigap:go` 실행 준비 완료 처리

### 가정

- `.gap/feature-login/analysis/gap-report.md`에는 로그인 요구사항, 현재 구현 요약, 갭, 위험, 검증 필요 사항이 정리되어 있습니다.
- 기존 프로젝트에는 로그인 UI 또는 인증 관련 진입점이 이미 일부 존재합니다.
- 새 인증 방식을 도입하기보다 기존 프로젝트 패턴을 우선 유지합니다.
- 실제 계획 파일을 작성할 때는 `.gap/feature-login/normalized/source-packet.md`와 `.gap/feature-login/analysis/gap-report.md`를 먼저 읽어 확인된 요구사항과 가정을 분리해야 합니다.

### 의존성

- 갭 보고서의 요구사항 및 non-goal
- 기존 인증 API 계약
- 라우팅/보호된 페이지 접근 정책
- 상태 관리 방식
- 에러 메시지/폼 검증 관례
- 테스트 도구 및 기존 테스트 패턴

### 현재 구현 확인 항목

실제 산출물을 작성하기 전 다음 구조를 확인해야 합니다.

- 로그인 페이지 또는 로그인 폼 컴포넌트 위치
- 인증 API 클라이언트 또는 service/module 위치
- 세션, 토큰, 사용자 상태 저장 방식
- 로그인 성공 후 이동 경로
- 인증 실패 시 표시되는 에러 처리 방식
- 기존 form validation 패턴
- 테스트 파일 위치와 naming convention

## 작업 분해 초안

아래 항목은 승인 전 계획 초안이며 실행 작업 목록으로 확정하지 않습니다.

### 작업 1. 갭 보고서 요구사항 매핑

- 목적: 갭 보고서의 요구사항, 구현 갭, 검증 필요 사항을 구현 단위로 분류합니다.
- 확인할 파일:
  - `.gap/feature-login/normalized/source-packet.md`
  - `.gap/feature-login/analysis/gap-report.md`
- 변경 가능성이 있는 파일: 없음
- 구현 메모: 확인된 요구사항과 가정을 분리하고, 불명확한 항목은 질문으로 남깁니다.
- 검증 방법: 모든 갭 항목이 구현 후보, non-goal, 질문 중 하나로 매핑되었는지 확인합니다.
- 위험: 보고서 내용이 부족하면 잘못된 구현 범위가 설정될 수 있습니다.

### 작업 2. 로그인 관련 코드 구조 조사

- 목적: 기존 로그인/인증 흐름의 책임 경계를 파악합니다.
- 확인할 파일:
  - 로그인 페이지/컴포넌트
  - 인증 API client 또는 service
  - 세션/사용자 상태 store 또는 context
  - 라우팅/리다이렉트 설정
  - 관련 테스트 파일
- 변경 가능성이 있는 파일: 없음
- 구현 메모: 새 패턴을 만들기 전에 기존 네이밍, 상태 관리, 에러 처리, 테스트 방식을 확인합니다.
- 검증 방법: 변경 후보 파일과 변경하지 않을 파일을 구분해 계획에 반영합니다.
- 위험: 인증 흐름이 여러 계층에 흩어져 있으면 변경 범위가 커질 수 있습니다.

### 작업 3. 로그인 제출 및 검증 흐름 보완 계획

- 목적: 입력 검증, 제출, 로딩, 중복 제출 방지 흐름을 요구사항에 맞춥니다.
- 확인할 파일:
  - 로그인 폼 컴포넌트
  - validation 유틸 또는 schema
  - UI 상태 처리 컴포넌트
- 변경 가능성이 있는 파일:
  - 로그인 폼/페이지 파일
  - validation 관련 파일
  - 관련 테스트 파일
- 구현 메모: 기존 폼 처리 패턴을 유지하고, 클라이언트 검증과 서버 검증 실패 표시를 구분합니다.
- 검증 방법: 빈 값, 잘못된 형식, 제출 중 상태, 중복 클릭 시나리오를 테스트합니다.
- 위험: UI 검증이 API 검증과 불일치하면 사용자에게 다른 오류가 표시될 수 있습니다.

### 작업 4. 인증 API 연동 및 실패 처리 계획

- 목적: 로그인 요청/응답 처리를 기존 API 계약과 맞춥니다.
- 확인할 파일:
  - 인증 API 호출부
  - API error normalization 유틸
  - session/token 저장 로직
- 변경 가능성이 있는 파일:
  - 인증 service/client
  - 로그인 submit handler
  - error handling 유틸
  - 관련 테스트 파일
- 구현 메모: 네트워크 오류, 인증 실패, 예기치 않은 응답을 구분해 처리합니다.
- 검증 방법: 성공 응답, 401/403, validation error, 네트워크 실패 케이스를 확인합니다.
- 위험: 토큰 저장 또는 세션 갱신 방식 변경은 보안 및 기존 인증 상태에 영향을 줄 수 있습니다.

### 작업 5. 로그인 성공 후 상태 갱신 및 이동 계획

- 목적: 로그인 성공 시 사용자 상태와 라우팅이 요구사항대로 동작하도록 합니다.
- 확인할 파일:
  - 사용자/session store 또는 context
  - route guard
  - 로그인 후 redirect 처리
- 변경 가능성이 있는 파일:
  - 인증 상태 관리 파일
  - 라우팅 설정 또는 guard
  - 로그인 페이지/handler
  - 관련 테스트 파일
- 구현 메모: 원래 접근하려던 URL 복귀, 기본 landing path, 이미 로그인한 사용자의 접근 처리 정책을 갭 보고서와 맞춥니다.
- 검증 방법: 로그인 성공 후 기본 이동, redirect 파라미터, 보호 페이지 접근 후 로그인 흐름을 검증합니다.
- 위험: 기존 라우팅 정책과 충돌하면 무한 리다이렉트가 발생할 수 있습니다.

### 작업 6. 테스트 및 수동 검증 정리

- 목적: 로그인 기능 변경에 필요한 자동/수동 검증을 명확히 합니다.
- 확인할 파일:
  - 기존 테스트 설정
  - 로그인/인증 관련 테스트
  - 프로젝트 검증 명령 문서 또는 설정
- 변경 가능성이 있는 파일:
  - 로그인 컴포넌트 테스트
  - 인증 service 테스트
  - 라우팅/통합 테스트
- 구현 메모: 프로젝트에 테스트 체계가 없다면 최소 수동 검증 체크리스트를 별도로 둡니다.
- 검증 방법: 성공/실패/로딩/redirect/세션 유지 케이스를 포함합니다.
- 위험: 인증 기능은 환경 의존성이 커서 mock과 실제 환경 검증 결과가 다를 수 있습니다.

## 검증 계획 초안

- 정적 검증: 프로젝트 lint/typecheck 명령이 있다면 실행
- 단위 검증: validation, API 응답 처리, error normalization
- 컴포넌트 검증: 입력 오류, 제출 중 UI, 실패 메시지, 성공 callback
- 통합 검증: 로그인 성공 후 세션 갱신 및 redirect
- 수동 검증:
  - 올바른 계정으로 로그인 성공
  - 잘못된 비밀번호로 실패 메시지 표시
  - 네트워크 실패 시 복구 가능한 안내 표시
  - 로그인 중 버튼 비활성화 또는 중복 제출 방지
  - 보호 페이지 접근 후 로그인하면 의도한 위치로 이동

## 롤백 계획 초안

- 로그인 UI 변경과 인증 service 변경을 가능한 한 분리합니다.
- 세션 저장 방식 변경이 필요하면 별도 작업으로 격리합니다.
- 문제가 생기면 submit handler/API 연동 변경부터 되돌리고 UI-only 변경은 독립적으로 판단합니다.
- redirect 정책 변경은 route guard 영향이 크므로 별도 커밋 또는 별도 작업으로 관리합니다.

## 검토가 필요한 결정 사항

1. 로그인 성공 후 기본 이동 경로는 어디인가요?
2. 원래 접근하려던 보호 페이지로 복귀해야 하나요?
3. 인증 실패 메시지는 서버 메시지를 그대로 표시하나요, 표준 문구로 변환하나요?
4. 세션/토큰 저장 방식은 기존 구현을 그대로 유지하나요?
5. 테스트는 어느 수준까지 필수로 요구하나요?
