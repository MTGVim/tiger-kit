## Source Packet 요약

- 작업 ID: `login-failure-error-message-gap`
- 원천 자료 유형: 사용자 브리프
- 관련 원천 자료: “로그인 실패 시 사용자는 실패 이유와 다음 행동을 알아야 한다. 현재는 'error'만 뜬다.”

### 문제

로그인 실패 시 사용자에게 단순히 `error`만 표시되어, 실패 원인과 이후 조치 방법을 알 수 없습니다.

### 목표

로그인 실패 상황에서 사용자가 실패 이유를 이해하고 다음 행동을 선택할 수 있도록 명확한 안내 메시지를 제공해야 합니다.

### 확인된 요구사항

1. 로그인 실패 시 실패 이유를 사용자에게 알려야 합니다.
2. 로그인 실패 시 사용자가 취할 수 있는 다음 행동을 안내해야 합니다.
3. 현재처럼 `error`만 표시되는 상태는 요구사항을 충족하지 않습니다.

### 인수 조건

- 로그인 실패 화면 또는 메시지에서 단순 `error` 대신 사람이 이해할 수 있는 실패 안내가 표시됩니다.
- 안내 문구에는 실패 이유가 포함됩니다.
- 안내 문구에는 사용자가 다음에 할 수 있는 행동이 포함됩니다.

### 제약 / 제외 범위

- 제공된 자료만으로는 대상 플랫폼, 로그인 방식, 인증 API, 화면 위치, 문구 톤, 보안 정책은 확정되지 않았습니다.
- 현재 구현 코드 경로가 제공되지 않았으므로 실제 구현 파일이나 데이터 흐름은 확인하지 않았습니다.

## 갭 분석 보고서

### 요약

현재 로그인 실패 UX는 사용자에게 `error`만 보여 주는 수준으로, 실패 이유와 다음 행동을 알려야 한다는 요구사항과 명확한 차이가 있습니다. 요구사항은 방향성이 분명하지만, 어떤 실패 이유를 어떤 수준까지 노출할지와 어떤 다음 행동을 제공할지는 추가 결정이 필요합니다.

### 확인된 요구사항

- 로그인 실패 시 사용자는 실패 이유를 알아야 합니다.
- 로그인 실패 시 사용자는 다음 행동을 알아야 합니다.
- 표시 메시지는 단순 기술 문자열인 `error`가 아니라 사용자가 이해 가능한 안내여야 합니다.

### 누락된 요구사항

- 로그인 실패 유형별 메시지 기준이 없습니다.
  - 예: 잘못된 비밀번호, 존재하지 않는 계정, 잠긴 계정, 비활성 계정, 네트워크 오류, 서버 오류, 인증 만료 등
- 실패 유형별 다음 행동이 정의되지 않았습니다.
  - 예: 다시 시도, 비밀번호 재설정, 계정 찾기, 관리자 문의, 잠시 후 재시도 등
- 메시지 노출 위치와 UI 형태가 정의되지 않았습니다.
  - 예: 입력 필드 하단, 폼 상단 배너, 토스트, 모달 등
- 접근성 요구사항이 없습니다.
  - 예: 스크린 리더 알림, 포커스 이동, 색상 대비
- 로깅/분석 요구사항이 없습니다.
  - 예: 어떤 실패 사유가 발생했는지 내부 로그에는 남기되 사용자에게는 안전한 메시지만 표시

### 모호한 요구사항

- “실패 이유”를 어느 정도 구체적으로 보여 줄지 불명확합니다.
  - 보안상 “계정이 없음”과 “비밀번호가 틀림”을 구분해 노출하지 않아야 할 수 있습니다.
- “다음 행동”의 범위가 불명확합니다.
  - 단순 재시도 안내인지, 비밀번호 재설정/회원가입/고객센터 연결까지 포함하는지 결정이 필요합니다.
- 로그인 실패의 범위가 불명확합니다.
  - 일반 ID/PW 로그인만인지, 소셜 로그인, SSO, 2FA, 계정 잠금까지 포함하는지 확인이 필요합니다.
- 대상 사용자와 문구 톤이 정해지지 않았습니다.

### 충돌하는 정보

- 제공된 원천 자료 내에서 충돌하는 정보는 없습니다.

### 구현 갭

- 현재 동작은 `error`만 표시된다고 되어 있어, 사용자 친화적인 에러 메시지 매핑이 없거나 UI에 반영되지 않는 갭이 있습니다.
- 실제 코드 경로가 제공되지 않아 다음 항목은 확인하지 못했습니다.
  - 인증 API 에러 응답 구조
  - 프론트엔드 에러 처리 위치
  - 기존 i18n/문구 관리 방식
  - 테스트 존재 여부

### UX / UI 갭

- `error`는 사용자가 원인과 조치 방법을 이해하기에 부족합니다.
- 실패 메시지에 다음 행동 링크나 버튼이 포함되어야 할 가능성이 높습니다.
  - 예: “이메일 또는 비밀번호가 올바르지 않습니다. 다시 입력하거나 비밀번호를 재설정해 주세요.”
- 실패 후 사용자가 어디를 수정해야 하는지 알 수 있도록 입력 필드, 폼 상태, 포커스 처리 기준이 필요합니다.

### API / 데이터 갭

- 클라이언트가 실패 사유를 구분할 수 있는 API 에러 코드가 있는지 확인이 필요합니다.
- 보안 정책상 사용자에게 노출 가능한 에러 코드/메시지와 내부 로깅용 에러 코드를 분리해야 할 수 있습니다.
- 네트워크 오류와 인증 실패처럼 사용자 행동이 다른 오류를 구분할 수 있어야 합니다.

### 엣지 케이스

- 이메일/아이디가 존재하지 않는 경우
- 비밀번호가 틀린 경우
- 계정이 잠긴 경우
- 계정이 비활성화되었거나 인증되지 않은 경우
- 소셜 로그인 또는 SSO 실패
- 2단계 인증 실패
- 네트워크 오류 또는 서버 장애
- 너무 많은 로그인 시도로 인한 rate limit
- 세션 만료 후 재로그인 실패

### 위험

- 실패 이유를 지나치게 구체적으로 노출하면 계정 존재 여부를 추측할 수 있는 보안 문제가 생길 수 있습니다.
- 모든 실패를 동일 문구로 처리하면 사용자 요구사항인 “실패 이유”를 충분히 만족하지 못할 수 있습니다.
- API 에러 코드와 UI 메시지 매핑이 명확하지 않으면 재발하거나 일부 케이스에서 다시 `error`가 노출될 수 있습니다.
- i18n이 있는 서비스라면 한국어/영어 등 다국어 문구 누락 위험이 있습니다.

### 확인이 필요한 질문

1. 로그인 방식은 무엇인가요? ID/PW만 포함하나요, 소셜 로그인/SSO/2FA도 포함하나요?
2. 보안 정책상 사용자에게 계정 존재 여부나 잠금 사유를 구체적으로 알려도 되나요?
3. 실패 유형별로 API 에러 코드가 이미 구분되어 있나요?
4. 원하는 다음 행동은 무엇인가요? 재시도, 비밀번호 재설정, 계정 찾기, 고객센터 문의 중 무엇을 포함해야 하나요?
5. 에러 메시지는 어디에 표시해야 하나요? 폼 상단, 필드 하단, 토스트, 모달 중 선호가 있나요?
6. 기존 문구 스타일 또는 i18n 체계가 있나요?
7. 이 변경의 대상 화면이나 코드 경로가 있나요?
