현재 요청은 사용자가 작성한 브리프를 원천 자료로 볼 수 있으므로, 이 기준으로 갭 분석을 진행했습니다. 이 평가 실행에서는 실제 `.gap/` 산출물을 만들지 않고, 아래에 생성했을 산출물 요약을 함께 남깁니다.

## 원천 자료 정규화 요약

### 목표

로그인 실패 시 사용자가 실패 이유를 이해하고, 다음에 무엇을 해야 하는지 알 수 있도록 오류 경험을 개선한다.

### 문제

현재 로그인 실패 시 화면에 `error`만 표시되어 사용자가 실패 원인과 해결 행동을 알 수 없다.

### 확인된 요구사항

- 로그인 실패 상황에서 단순한 `error` 대신 실패 이유를 사용자에게 알려야 한다.
- 로그인 실패 상황에서 사용자가 취할 수 있는 다음 행동을 안내해야 한다.
- 안내 문구는 사용자가 이해할 수 있는 표현이어야 한다.

### 인수 조건

- 로그인 실패 시 화면에 `error`만 단독으로 표시되지 않는다.
- 로그인 실패 메시지는 실패 이유를 포함한다.
- 로그인 실패 메시지는 다음 행동을 포함한다.
- 사용자는 메시지만 보고 재시도, 입력 수정, 비밀번호 재설정, 지원 문의 등 적절한 다음 행동을 판단할 수 있다.

### 제약

- 제공된 원천 자료에는 구체적인 로그인 방식, 인증 API, 에러 코드, UI 위치, 지원 채널이 명시되어 있지 않다.
- 현재 구현 파일이나 화면 자료는 제공되지 않았다.

### 제외 범위

- 실제 구현 변경은 이 단계에서 수행하지 않는다.
- 로그인 정책, 계정 잠금 정책, MFA 정책 자체의 변경 여부는 아직 확정하지 않는다.

### 열린 질문

- 로그인 실패 이유를 어디까지 구체적으로 노출할 것인가?
- 보안상 사용자 존재 여부를 추론할 수 있는 문구를 피해야 하는가?
- 실패 유형은 어떤 것들이 있는가? 예: 잘못된 비밀번호, 존재하지 않는 계정, 잠긴 계정, 비활성 계정, 네트워크 오류, 서버 오류, 인증 만료 등.
- 비밀번호 재설정, 회원가입, 고객지원 등 다음 행동 링크가 이미 존재하는가?
- 문구는 어떤 톤앤매너와 언어 정책을 따라야 하는가?

## 갭 분석 보고서

### 요약

현재 요구는 “로그인 실패 시 `error`만 보이는 문제를 개선하여 실패 이유와 다음 행동을 제공한다”는 것입니다. 핵심 갭은 에러 표시가 지나치게 일반적이라 사용자가 문제를 해결할 수 없다는 UX 갭입니다. 다만 실패 유형, 보안 정책, 노출 가능한 상세 수준, 실제 현재 구현 위치가 아직 확인되지 않아 구현 계획 전 추가 확인이 필요합니다.

### 확인된 요구사항

- 로그인 실패 시 사용자에게 실패 이유를 알려야 한다.
- 로그인 실패 시 사용자에게 다음 행동을 알려야 한다.
- 현재의 `error` 단독 표시는 기준을 만족하지 않는다.
- 오류 메시지는 사용자가 이해 가능한 문장이어야 한다.

### 누락된 요구사항

- 실패 유형별 메시지 기준이 없다.
- 실패 유형별 다음 행동 매핑이 없다.
- 보안상 노출 가능한 실패 이유의 범위가 정의되지 않았다.
- 메시지 표시 위치, 디자인, 접근성 기준이 없다.
- 서버/API에서 내려오는 에러 코드와 프론트엔드 문구 매핑 기준이 없다.
- 로깅이나 모니터링 기준이 없다.
- 테스트 기준이 구체화되어 있지 않다.

### 모호한 요구사항

- “실패 이유”가 기술적 원인인지, 사용자 친화적 범주인지 불명확하다.
- “다음 행동”이 단순 문구인지, 링크/버튼까지 포함하는지 불명확하다.
- 모든 로그인 실패에 동일한 안내를 쓸지, 실패 유형별로 다르게 안내할지 불명확하다.
- `error`가 화면 문구인지, 토스트/알림/필드 에러/콘솔 로그인지 불명확하다.

### 충돌하는 정보

- 제공된 자료 안에서는 충돌하는 정보가 확인되지 않았다.

### 구현 갭

- 현재 구현 파일이 제공되지 않아 실제 코드 확인은 수행하지 못했다.
- 현재는 로그인 실패 응답이 사용자 친화적인 메시지로 변환되지 않고 `error`로 표시되는 것으로 보인다.
- 인증 실패 응답의 에러 타입 또는 상태 코드가 UI 메시지로 매핑되는 구조가 필요하다.
- 예상 구현 지점은 로그인 요청 처리, 에러 핸들링, 메시지 렌더링, 테스트 코드다.

### UX / UI 갭

- `error`는 사용자가 원인과 해결 방법을 이해하기에 부족하다.
- 입력값 오류, 계정 문제, 시스템 문제를 구분해 안내하지 않으면 사용자가 불필요하게 반복 시도할 수 있다.
- 다음 행동이 없으면 비밀번호 재설정, 입력 확인, 잠시 후 재시도, 고객지원 문의 같은 해결 경로가 막힌다.
- 에러 문구가 필드 주변에 표시되어야 하는지, 전역 알림으로 표시되어야 하는지 정해져 있지 않다.

### API / 데이터 갭

- 인증 API가 실패 원인을 구분 가능한 코드로 제공하는지 확인이 필요하다.
- API가 단순 문자열 `error`만 반환한다면, 프론트엔드에서 충분한 안내를 제공하기 어렵다.
- 에러 코드 예시와 메시지 매핑 표가 필요하다.
- 네트워크 오류와 인증 실패를 구분할 수 있어야 한다.

### 엣지 케이스

- 이메일/아이디 형식 오류
- 비밀번호 오입력
- 존재하지 않는 계정
- 비활성화 또는 잠긴 계정
- 비밀번호 재설정이 필요한 계정
- 네트워크 연결 실패
- 서버 장애
- 너무 많은 로그인 시도
- MFA 또는 추가 인증이 필요한 계정
- 세션 만료 후 재로그인 실패

### 위험

- 실패 이유를 너무 자세히 노출하면 계정 존재 여부나 보안 상태가 추론될 수 있다.
- 실패 유형을 과도하게 단순화하면 사용자가 실제 해결 행동을 찾지 못할 수 있다.
- API 에러 코드가 안정적이지 않으면 UI 문구가 잘못 표시될 수 있다.
- 다국어 제품이라면 문구 관리와 번역 누락 위험이 있다.
- 테스트 없이 변경하면 기존 로그인 실패 플로우가 깨질 수 있다.

### 확인이 필요한 질문

1. 로그인 실패 이유를 사용자에게 어느 수준까지 노출해도 되나요?
2. 현재 인증 API는 실패 유형별 에러 코드나 상태 코드를 제공하나요?
3. `error`가 표시되는 화면 또는 코드 경로는 어디인가요?
4. 비밀번호 재설정, 회원가입, 고객지원 링크가 이미 있나요?
5. 실패 유형별 문구를 별도로 만들까요, 아니면 보안상 일반화된 문구를 사용할까요?
6. 접근성 기준이나 디자인 시스템의 에러 메시지 컴포넌트가 있나요?

### 추천 다음 단계

1. 현재 로그인 구현 경로와 인증 API 응답 형식을 확인한다.
2. 실패 유형별 메시지와 다음 행동 매핑표를 정의한다.
3. 보안 정책에 맞춰 계정 존재 여부가 노출되지 않도록 문구 수준을 결정한다.
4. `/tigap:gaplan`으로 구현 계획과 작업 목록을 작성한다.

## 생성했을 산출물 요약

평가 지시에 따라 실제 `.gap/` 파일은 만들지 않았습니다. 일반 실행이었다면 다음 파일을 작성했을 것입니다.

- `.gap/{branch_name}/normalized/source-packet.md`: 위 “원천 자료 정규화 요약” 내용을 기반으로 목표, 문제, 요구사항, 인수 조건, 제약, 열린 질문을 정리
- `.gap/{branch_name}/analysis/gap-report.md`: 위 “갭 분석 보고서” 내용을 템플릿에 맞춰 정리

다음 단계로는 `/tigap:gaplan`을 실행해 이 갭 분석을 구현 계획과 작업 목록으로 바꾸는 것을 권장합니다.
