# gap

## 목적

흩어진 원천 자료를 구체적인 갭 분석 보고서로 바꿉니다.

계획이나 구현을 시작하기 전에 무엇이 빠졌는지, 모호한지, 충돌하는지, 위험한지, 아직 구현되지 않았는지 파악해야 할 때 이 스킬을 사용합니다.

## 동작 방식

사용자가 명시적으로 워크플로우를 건너뛰라고 하지 않는 한 이 스킬에서는 코드를 구현하지 않습니다.

다음에 집중합니다.

1. 자료 수집
2. 자료 정규화
3. 중복 제거
4. 현재 상태 확인
5. 갭 분석
6. 명확한 다음 단계 추천

## 입력 자료

source-of-truth 자료가 아직 없다면 분석을 시작하지 않고 사용자에게 요청합니다.

자료가 없고 사용자가 자료를 가져오거나 추출하라고도 하지 않았다면 다음을 수행합니다.

1. 아직 gap report를 만들 수 없다고 알립니다.
2. source-of-truth로 사용할 수 있는 자료 예시를 보여줍니다.
3. 사용자에게 다음 중 하나를 선택하게 합니다.
   - 자료 링크나 경로를 직접 제공
   - 이슈, PR, 문서, 코드 경로 등에서 자료를 가져오라고 지시
   - 인터뷰 방식으로 요구사항 수집 시작

이 경우 저장소를 임의로 source-of-truth로 삼아 분석하지 않고, `.gap/` 산출물도 생성하지 않습니다.

허용되는 자료 유형은 다음과 같습니다.

- 이슈 트래커 티켓
- 지식베이스 문서
- PRD
- 디자인 문서
- 사용자가 작성한 브리프
- 스크린샷
- 코드 경로
- 풀 리퀘스트
- 기존 구현 참고 자료
- 일반 텍스트 메모

특정 도구 사용을 강제하지 않습니다. 모든 외부 시스템은 범용 자료 제공처로 취급합니다.

## 산출물 디렉터리

가능하면 현재 git 브랜치 이름을 사용합니다.

산출물은 다음 경로 아래에 저장합니다.

```text
.gap/{branch_name}/
```

브랜치 이름을 알 수 없다면 짧은 작업 ID를 사용자에게 묻거나 작업 제목에서 안전한 slug를 추론합니다.

## 필수 출력 파일

다음 파일을 생성하거나 갱신합니다.

```text
.gap/{branch_name}/normalized/source-packet.md
.gap/{branch_name}/analysis/gap-report.md
```

필요하면 다음 디렉터리도 생성합니다.

```text
.gap/{branch_name}/sources/
```

추적 가능성에 도움이 될 때만 자료 스냅샷이나 요약을 저장합니다.

## 진행 절차

### 1. 자료 참조 수집

자료나 자료 추출 지시가 없다면 여기서 멈추고 다음 안내를 제공합니다.

```text
Gap 분석을 시작하려면 source-of-truth가 필요합니다.
다음 중 하나를 제공하거나 선택해 주세요.

1. 이슈 트래커 티켓, 문서, PRD, 디자인, 브리프, 스크린샷, 코드 경로
2. "GitHub issues를 가져와", "이 PR을 기준으로 분석해", "README와 현재 구현을 비교해" 같은 자료 추출 지시
3. 인터뷰 방식으로 요구사항 정리 시작
```

사용자가 인터뷰를 선택하면 최소한 다음을 질문합니다.

```text
- 해결하려는 문제는 무엇인가요?
- 원하는 완료 상태는 무엇인가요?
- 반드시 지켜야 할 제약이나 제외할 범위가 있나요?
- 참고해야 할 기존 구현, 문서, 화면, 이슈가 있나요?
```

답변만으로 source packet의 Problem, Goal, Confirmed Requirements, Constraints, Non-goals를 채울 수 있을 때 다음 단계로 진행합니다.

### 2. Source Packet으로 정규화

모든 자료를 source packet 템플릿으로 변환합니다.

다음을 기록합니다.

- 목표
- 문제
- 확인된 요구사항
- 인수 조건
- 제약
- 논의 범위에서 제외할 것
- 결정 사항
- 코멘트 또는 논의 지점
- 열린 질문
- 관련 자료

### 3. 중복 제거

두 자료가 같은 내용을 반복하면 하나의 대표 문장만 남기고 중복되었다는 사실을 기록합니다.

두 자료가 충돌하면 조용히 합치지 말고 충돌을 명시합니다.

### 4. 현재 구현 확인

코드에 접근할 수 있다면 구현 갭을 작성하기 전에 관련 파일을 확인합니다.

다음을 살펴봅니다.

- 기존 동작
- 관련 컴포넌트
- 기존 데이터 흐름
- 검증 규칙
- 테스트
- feature flag
- 에러 처리
- 네이밍 패턴

### 5. 갭 분석 보고서 작성

`templates/gap-report.md`를 사용합니다.

발견 사항은 다음으로 분류합니다.

- 확인된 요구사항
- 누락된 요구사항
- 모호한 요구사항
- 충돌하는 정보
- 구현 갭
- UX/UI 갭
- API/데이터 갭
- 엣지 케이스
- 위험
- 확인이 필요한 질문

## 완료 기준

이 스킬은 다음 조건을 만족하면 완료됩니다.

- 자료가 정규화됨
- 중복과 충돌 정보가 처리됨
- 가능한 범위에서 현재 구현을 확인함
- 갭 분석 보고서가 존재함
- `/tigap:gaplan`으로 넘기기에 충분히 다음 단계가 명확함

## 인계

갭 분석 보고서가 준비된 뒤에만 다음 명령을 추천하며 마무리합니다.

```text
/tigap:gaplan
```
